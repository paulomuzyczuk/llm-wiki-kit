# Contributing

Thanks for your interest. This is a small, opinionated project; the goal of this
guide is to make a good contribution easy to get right on the first try.

Before you start anything non-trivial, **open an issue** describing the change. It's a
solo-maintained project, so a quick check that a change fits the design saves everyone
a wasted PR. Small fixes (typos, a clearly-broken thing) can go straight to a PR.

By participating you agree to keep things respectful and constructive. Security issues
should **not** be filed as public issues — see [`SECURITY.md`](SECURITY.md).

## What contributions fit

- **Tooling** — the Python/shell utilities (`lint.py`, `check-conformance.py`,
  `new-vault.sh`, `install.sh`) and CI.
- **Skills** — the Claude Code skills under `book-*/`, `article-ingestion/`,
  `vault-evaluator/`, `vault-lint/`.
- **The contract** — `CLAUDE.template.md`. Changes here are the most sensitive; see
  [The contract](#the-contract-claudetemplatemd) below.
- **Docs** — README, THEORY, this file.

The populated example vault under `examples/software-craft/` is a curated showcase
built by running the real pipeline. Please don't hand-edit it casually — it must stay
lint-clean (see below), and its content reflects actual ingested sources.

## Development setup

No build step and no third-party runtime dependencies — just Python 3.10+.

```sh
git clone https://github.com/paulomuzyczuk/llm-wiki-kit.git
cd llm-wiki-kit
./install.sh            # optional: symlink the skills into ~/.claude/skills
```

For the dev tooling (linter/formatter/security), install:

```sh
pip install ruff bandit
# shellcheck via your package manager, e.g. `brew install shellcheck`
```

## Running the checks locally

CI runs three jobs (`lint`, `security`, `test`). Reproduce them before pushing:

```sh
# lint
ruff check .
ruff format --check .
shellcheck install.sh new-vault.sh */regenerate-clones.sh

# security
bandit -r check-conformance vault-lint -x '*/tests/*' --severity-level medium

# test
python3 -m pytest -q check-conformance/tests/
python3 -m pytest -q vault-lint/tests/

# the example vault must stay lint-clean (Phase 1 findings: 0)
python3 vault-lint/lint.py examples/software-craft

# a freshly scaffolded vault must pass conformance with exit 0
./new-vault.sh --subs check-conformance/subs/software-craft.json --dest /tmp/ci-vault
python3 check-conformance/check-conformance.py \
  --vault /tmp/ci-vault/CLAUDE.md --template CLAUDE.template.md \
  --subs check-conformance/subs/software-craft.json
```

Formatting is owned by `ruff format` (config in `ruff.toml`, single-quote style); run
it rather than hand-formatting. `E501` line-length is intentionally left to the
formatter, so don't hand-wrap long strings or comments.

## Editing a skill

`SKILL.md` is the **canonical source** for each skill. The `<skill>-skill.md` file is a
**derived clone** (a project-context upload copy) regenerated by the skill's
`regenerate-clones.sh` — it's git-ignored and must never be edited directly. After
changing a `SKILL.md`, run that skill's `./regenerate-clones.sh` if you rely on the
clone locally.

## The contract (`CLAUDE.template.md`)

`CLAUDE.template.md` is the single source of truth that every vault's `CLAUDE.md` is
instantiated from, and `check-conformance.py` enforces that vaults don't drift from it
outside the **sanctioned-divergence regions**. If you change the template:

- Update the example subs (`check-conformance/subs/software-craft.json`) and the
  conformance tests if token sets or regions change.
- Confirm the scaffolder round-trip still passes (the `new-vault.sh` →
  `check-conformance.py` sequence above).

When in doubt about whether something is allowed to differ, read the instantiation
header at the top of the template and the sanctioned-region markers.

## Commits and pull requests

- **Conventional Commits**, with a scope where it helps:
  `feat(vault-lint): …`, `fix: …`, `docs: …`, `test: …`, `ci: …`, `style: …`.
- Keep PRs **focused** — one logical change. Separate mechanical reformatting from
  behavior changes (e.g. a `style:` commit apart from a `feat:` commit).
- Branch from `main`; make sure **CI is green** before requesting review.
- Update [`CHANGELOG.md`](CHANGELOG.md) under `## [Unreleased]` for user-facing changes.
- If your change affects how something is used, update the README in the same PR.

## Adding example content

The example vault is produced by the real planner → ingestion → review pipeline rather
than written by hand, so it stays faithful to the system. If you do touch it, the
non-negotiable bar is that `python3 vault-lint/lint.py examples/software-craft` reports
`Phase 1 findings: 0`.
